# 01_intake.md

## Company/client context
Client is Tony (personal project). Project is a personal website rebuild.

## Problem framing (requested vs underlying need)
- Requested: build a terminal-first personal website with a clean companion web experience.
- Underlying need: create a memorable, high-signal personal brand asset for recruiters/hiring managers and other visitors.

## Desired outcomes
1. Visitors quickly understand who Tony is and what he has done.
2. Experience feels distinctive and non-boring.
3. Contact and resume access are frictionless.
4. Terminal-first brand is preserved without losing basic accessibility.

## Stakeholders and decision owner
- Decision owner: Tony
- Primary audience: recruiters/hiring managers
- Secondary audience: peers and social visitors

## Constraints (timeline, budget, compliance, dependencies)
- Target: use this as software-factory pilot and move quickly.
- MVP focus: desktop-first; mobile is nice-to-have but not primary.
- Architecture direction: terminal-first interaction model + minimal web companion.

## Initial scope assumptions (non-binding)
- Final sections:
  1. Hero
  2. Product Manager
  3. AI Product Builder
  4. Projects
  5. Contact
- Resume should be downloadable and accessible from both web and terminal flows.

## Newly confirmed UI/UX direction
- Hero/landing should look like a terminal CLI.
- Use ASCII-art profile rendering in hero (inspired by reference example).
- Navigation should be command-driven (terminal commands control section routing).
- Minimal web companion should still expose core profile info for quick recruiter scan.

## Risks / unknowns
1. ASCII profile quality/performance tradeoff (size, readability, loading).
2. Command UX discoverability for non-technical visitors.
3. Risk of over-indexing on style over content clarity.

## Recommendation
- **Proceed** to PRD/UX requirements lock.
- Why: enough direction is now confirmed to produce detailed requirements and start blueprint.
